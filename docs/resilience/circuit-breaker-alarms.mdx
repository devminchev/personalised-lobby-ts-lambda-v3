# Circuit-breaker alarms

The `PostCloudWatchController` Lambda maintains a single global circuit-breaker state (`BREAKER#GLOBAL` in DynamoDB) driven entirely by CloudWatch alarm state-change events. This document describes **what the alarms are, what they measure, how fast they respond, and what their ALARM/OK state means.** It is the spec the controller's callers and operators can rely on.

The stack emits two categories of alarm:

- **Signal alarms** (`breaker:*`-prefixed) — consumed by the controller, drive breaker state. The majority of this document is about these.
- **Self-observability alarms** (not `breaker:*`-prefixed) — watch the controller itself. Never consumed by the controller; exist so operators can notice when the controller fails. Covered in a dedicated section near the bottom.

For the controller's state machine itself (CLOSED → OPEN → HALF_OPEN), see the function's README at `functions/PostCloudWatchController/`.

## Design overview

The breaker reacts to two classes of signal:

| Class | Behaviour on a single signal | Threshold philosophy |
|---|---|---|
| **Critical** | One signal in ALARM opens the breaker immediately | Rare, unambiguous "stop hammering the backend now" signals |
| **Warning** | One signal in ALARM → `single_warning`, breaker stays CLOSED. Two or more simultaneously → `compound_warning`, breaker opens. | Pressure signals that are individually noisy but meaningful in combination |

The routing is done by the alarm **name**. Any CloudWatch alarm whose name starts with `breaker:` is consumed by the controller via an EventBridge rule; anything else is invisible to the breaker. The name format is one of:

- `breaker:GLOBAL:<signal>` — 3-part, used when there is no useful human label to attach
- `breaker:GLOBAL:<label>:<signal>` — 4-part, used when a display-only label helps console/ticket searches (e.g. the API's `ServiceName`)

`GLOBAL` is a fixed routing key — all signals write to the shared `BREAKER#GLOBAL` partition regardless of the label. The `<signal>` segment (always the trailing `:`-separated token) is what maps to the critical/warning classification in `functions/PostCloudWatchController/src/config.ts`.

## Signal catalogue

### Critical signals

#### `breaker:GLOBAL:os-search-rejections`
#### `breaker:GLOBAL:os-write-rejections`

**What they measure:** the number of OpenSearch thread-pool rejections on the domain's search / write pools, per minute.

**Why metric math (DIFF):** AWS publishes `AWS/ES ThreadpoolSearchRejected` and `ThreadpoolWriteRejected` as **monotonic lifetime counters** — each datapoint is the total number of rejections the node has seen since it started, not the per-period rate. A plain `Statistic: Sum` alarm therefore returns the lifetime total every minute and never recovers once the domain has ever had a rejection burst. The alarms instead use metric math:

```
rejectionDelta = DIFF(counter)
```

where `counter` is `Sum(AWS/ES Threadpool{Search,Write}Rejected)` over a 60-second window. `DIFF` returns "new rejections in the last period", which is what the threshold of `5` is meant to measure.

Node restarts briefly produce a large negative `DIFF` (the counter resets to 0); this is below threshold so the alarm stays OK. No false open.

**Detection timing:** `Period: 60`, `EvaluationPeriods: 3`, `DatapointsToAlarm: 2`.

- Best case (sustained breach, 2 consecutive bad datapoints): **2 min**
- Worst case (2-of-3 sliding): **3 min**

**Missing data:** `TreatMissingData: breaching`. A stall in OpenSearch metric publication (domain outage, IAM glitch) is itself a severe signal — we want the breaker to open, not sit blind.

### Warning signals

#### `breaker:GLOBAL:jvm-pressure`
#### `breaker:GLOBAL:cpu-high`

**What they measure:** the `JVMMemoryPressure` and `CPUUtilization` metrics AWS publishes per node under `AWS/ES`.

**Statistic and dimensions:** `Statistic: Maximum` against the domain-level dimension (`DomainName + ClientId`, no `NodeId`). AWS publishes both per-node and domain-level entries; querying at the domain level with `Maximum` returns the hottest-node/hottest-minute reading in the evaluation period. This catches a single hot node that a cluster-average would hide.

**Thresholds:**
- `jvm-pressure`: 85 % — just below the heap ratio where JVM GC pauses start to hurt query latency.
- `cpu-high`: 80 % — sustained CPU saturation on a search cluster degrades p99 well before the JVM is in trouble, so the CPU threshold is set below JVM's.

**Detection timing:** `Period: 60`, `EvaluationPeriods: 5`, `DatapointsToAlarm: 3`.

- Best case (sustained breach, 3 consecutive bad minutes): **3 min**
- Worst case (3-of-5 sliding): **5 min**

**Missing data:** `TreatMissingData: notBreaching`. No traffic and stalled metrics look the same for these signals, and a false breach on a warning would over-trigger the compound path.

#### `breaker:GLOBAL:<ServiceName>:api-p99`
#### `breaker:GLOBAL:<ServiceName>:api-5xx`

**What they measure:** API Gateway end-to-end latency p99 and 5xx error ratio for the `ServiceName` (currently `lobby-v2`) stage.

**Why metric math (count floor):** at low traffic, both of these metrics collapse into individual request outliers. A P99 computed over 1–2 requests is just "the slowest one"; a 5xx `Average` with 2 requests where one is a 5xx reads as "50 % error rate". To suppress those false signals, each alarm's result expression is gated by a traffic floor:

```
p99Gated    = IF(requestCount >= 100, p99Latency, 0)
errorRatio  = IF(requestCount >= 100, errorCount / requestCount, 0)
```

When `requestCount` (the `Sum(AWS/ApiGateway Count)` over the same window) is below 100 the expression returns 0, which is below every threshold, so the alarm stays OK regardless of how slow or how many 5xx any of those few requests were. When the floor is met, the alarm behaves identically to a plain threshold check.

**Thresholds:**
- `api-p99`: 5000 ms.
- `api-5xx`: 0.01 (1 % error ratio).

**Detection timing (as currently tuned):** `Period: 300`, `EvaluationPeriods: 5`, `DatapointsToAlarm: 3`.

- Best case (sustained breach): **15 min**
- Worst case (3-of-5 sliding): **25 min**

> **Known asymmetry with the OS warning signals.** The JVM / CPU warnings were retuned to `Period: 60` to keep warning detection faster than critical (2–3 min) so the tiered design works — warnings catch trouble *before* critical fires. The API warnings have **not** been retuned yet because their count-floor semantic is sensitive to period choice (`100 req / 5 min = 0.33 rps` floor vs `100 req / 1 min = 1.67 rps` floor), and the right floor depends on production traffic patterns that have not been observed yet. The practical effect today: an `api-p99 + api-5xx` compound trip takes 15–25 min; any compound involving a JVM/CPU signal is gated by the slower API partner. Revisit once real traffic data is available.

**Missing data:** `TreatMissingData: notBreaching`, for the same reason as JVM/CPU.

## Detection timing at a glance

| Alarm | Period | EvalPeriods | DPToAlarm | Best-case trip | Worst-case trip |
|---|---|---|---|---|---|
| `os-search-rejections` (critical) | 60 s | 3 | 2 | 2 min | 3 min |
| `os-write-rejections` (critical) | 60 s | 3 | 2 | 2 min | 3 min |
| `jvm-pressure` (warning) | 60 s | 5 | 3 | 3 min | 5 min |
| `cpu-high` (warning) | 60 s | 5 | 3 | 3 min | 5 min |
| `api-p99` (warning) | 300 s | 5 | 3 | 15 min | 25 min |
| `api-5xx` (warning) | 300 s | 5 | 3 | 15 min | 25 min |

"Best case" = metric goes bad and stays bad continuously. "Worst case" = metric breaches intermittently and needs to accumulate `DatapointsToAlarm` bad readings within the sliding `EvaluationPeriods` window.

## State-transition semantics

CloudWatch alarms publish EventBridge events **only on state transitions** (`OK ↔ ALARM ↔ INSUFFICIENT_DATA`), never on every evaluation. A sustained incident therefore produces exactly **two events** — one when the alarm trips, one when it recovers — regardless of how long the bad condition lasts or how short `Period` is.

This is why tightening `Period` from 300 s to 60 s on the OS warnings does not make the system chatty. It speeds detection up, but the steady-state event volume is bounded by the **number of transitions**, not the number of evaluations.

## Anti-flap

Two protections sit between the raw metric and the breaker state seen by consumers:

1. **`DatapointsToAlarm` smoothing.** All alarms require multiple bad datapoints in the last `EvaluationPeriods` window — a single-minute GC pause or a burst health check does not trip a warning.
2. **Controller-side `openHoldMs: 60_000`.** Once the breaker is OPEN, it stays OPEN for at least 60 s regardless of subsequent alarm state. Downstream consumers cannot observe the breaker state whip-saw faster than once per minute, even if the underlying alarm flaps.

The combination means the effective anti-flap window seen by callers is **max(alarm smoothing, 60 s)**.

## Missing-data policy

| Signal class | `TreatMissingData` | Rationale |
|---|---|---|
| Critical (OS rejections) | `breaching` | A stall in ES metrics is itself a severe signal — the breaker should open, not stay blind. |
| Warning (JVM, CPU, API) | `notBreaching` | "No traffic" and "metrics stalled" look the same here; a missing-data breach on a warning would over-trigger the compound path. |

## Controller self-observability alarms

Two alarms watch the **controller Lambda itself** — its error rate and the depth of its async-failure DLQ. They are deliberately **not** breaker signals: their names do not start with `breaker:`, the EventBridge rule does not route them, and the controller never sees them. They exist so operators can notice when the controller itself has failed — not to change what the breaker does.

These alarms don't write to DynamoDB, don't participate in signal aggregation, compound scoring, or state-machine transitions. Their only consumers are the CloudWatch console, a future dashboard, and (when it lands) an SNS → paging pipeline.

### `lobby-controller-errors-${StageName}`

| Property | Value |
|---|---|
| Metric | `AWS/Lambda Errors` on `PostCloudWatchControllerFunction` |
| Threshold | `≥ 1` |
| Period / EvaluationPeriods / DatapointsToAlarm | `60 s` / `5` / `3` |
| Detection window | 3 min best case, 5 min worst case |
| Missing data | `notBreaching` |

Threshold is `1` because the controller is invoked rarely (alarm state changes + the 60 s tick) and there's no legitimate "error budget". The 3-of-5 smoothing exists to filter a single Lambda auto-retry before paging — not to tolerate ongoing errors. If the retry also fails, the DLQ alarm below fires.

### `lobby-controller-dlq-depth-${StageName}`

| Property | Value |
|---|---|
| Metric | `AWS/SQS ApproximateNumberOfMessagesVisible` on `PostCloudWatchControllerDeadLetterQueue` |
| Threshold | `≥ 1` |
| Period / EvaluationPeriods / DatapointsToAlarm | `60 s` / `1` / `1` |
| Detection window | Fires on the first message |
| Missing data | `notBreaching` |

No smoothing is applied because a message in the DLQ means Lambda has **already** exhausted its async-invocation retries (initial attempt + 2 auto-retries = 3 failures). By the time the metric reports `≥ 1`, an alarm state-change event has been permanently lost to the controller. The appropriate response is immediate operator attention, not a 3-of-5 delay.

### Relationship between the two

Both observe controller health at different depths:

| Alarm | Fires when | Severity |
|---|---|---|
| Errors | The controller is throwing exceptions *right now*, faster than Lambda auto-retry can cover | Important — may be transient |
| DLQ depth | The controller has already failed enough times on one event that Lambda gave up | Permanent loss of that event — always an incident |

Errors is the early-warning signal; DLQ depth is the "something has already gone wrong past the point of recovery" signal. Both of them firing at the same time usually means "the errors have been sustained long enough to overwhelm Lambda's retry budget".

### Why they are not `breaker:`-prefixed

Naming either of these alarms `breaker:...` would couple them to the controller's own EventBridge rule. A controller error → this alarm fires → EventBridge invokes the controller → the controller may still be broken, fails again → alarm stays in ALARM — a feedback loop that amplifies the original failure and consumes the retry budget. The name constraint is the *only* mechanism preventing that loop; neither the state machine nor the EventBridge rule can otherwise tell apart "an alarm describing my own health" from "an alarm describing backend health".

## Invariants the controller relies on

- **Alarm name is the only routing key.** The `<signal>` suffix of the alarm name is matched against `CRITICAL_SIGNALS` / `WARNING_SIGNALS` in `src/config.ts`. Adding a new alarm without updating that file means the controller will receive the event and ignore it.
- **Only `breaker:*`-prefixed alarms are consumed.** The EventBridge rule pattern is `alarmName prefix: "breaker:"`. Any other alarm in the account is invisible to the breaker. The controller self-observability alarms above (`ControllerErrorsAlarm`, `ControllerDlqDepthAlarm`) MUST NOT be renamed with the `breaker:` prefix — see the section above for the feedback-loop reasoning.
- **`GLOBAL` is the only partition the controller acts on.** Per-service routing is intentionally collapsed — see `parseAlarmName` in `src/index.ts` for the full reasoning.

## Tuning levers

When a tuning change is needed, these are the knobs and their implications:

| Knob | What it controls | Trade-off |
|---|---|---|
| `Threshold` | How bad is "bad" | Lower = more sensitive, more false trips. Keep thresholds dimensional (rejections/min, %, ms) so the number is meaningful independent of traffic. |
| `Period` | Sample-aggregation granularity | Smaller = faster detection, more sensitivity to short spikes. Combined with `DatapointsToAlarm` smoothing, small periods remain robust. |
| `EvaluationPeriods × Period` | Worst-case detection window | Longer = more tolerant of intermittent breaches, slower detection. |
| `DatapointsToAlarm` | How many bad datapoints in the window are needed | Higher = more smoothing, slower detection. `DatapointsToAlarm / EvaluationPeriods` ratio is the filter strength. |
| `TreatMissingData` | Fail-open vs fail-closed on absent data | `breaching` for signals where absence is itself an incident; `notBreaching` for signals where absence just means quiet. |

When changing any of these, update the inline comment in `template.yaml` and the detection-timing table above so the spec stays in sync with reality.
