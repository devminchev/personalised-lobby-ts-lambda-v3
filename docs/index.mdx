# Personalised Lobby Ts Lambda V3

A collection of AWS Lambda functions and shared layers that provide the backend functionality required by the frontend for lobby personalisation. Lambdas are organised as independent packages, built with TypeScript, and deployed via AWS SAM on dev and as docker images on STG and PROD.

## Repository Structure

```
personalised-lobby-ts-lambda-v3/
  functions/        Individual lambda packages (one per API endpoint)
  libs/             Shared libraries packaged as lambda layers (e.g. OSClient)
  events/           Sample invocation events for local testing
  template.yaml     SAM template defining all AWS resources
  scripts/          Build, deploy, and CI helper scripts
```

Each lambda under `functions/` is a self-contained package with its own `package.json`, tests, and README. Examples include `GetNavigationFunction`, `GetGamesFunction`, `GetRecentlyPlayedFunction`, `GetPersonalisedSectionGamesFunction`, and many more covering games, sections, navigation, recommendations, and personalised data.

## Tech Stack

- **Runtime:** Node.js 20 on AWS Lambda (arm64 by default, x86_64 supported)
- **Language:** TypeScript, bundled with esbuild
- **Infrastructure:** AWS SAM (CloudFormation) with API Gateway
- **Monorepo orchestration:** NX
- **Testing:** Jest (per-lambda `tests/` folders)
- **Containers:** Multi-stage Docker builds (Node.js builder + RHEL9 Node.js 20 runtime)
- **Search backend:** OpenSearch via the shared `OSClient` layer

## Getting Started

The top-level [README](../README.md) covers the full setup: installing the AWS CLI, SAM CLI, Node.js 20, and Docker; configuring AWS SSO against the `lobby-playground` profile; and scaffolding a new lambda from the shared `kickstart` template.

Once dependencies are installed, the typical local workflow is:

```bash
sam build
sam local invoke "GetNavigationFunction" \
  -e functions/GetNavigationFunction/events/event.json \
  --env-vars env.json
```

For a single function or layer, scope the build with `--parameter-overrides "FunctionName=... LayerName=... LambdaArchitecture=..."`.

## Docker

Lambda images can be built and run locally through NX targets (`yarn docker:build`, `yarn nx docker <function-name>`, `yarn nx run-docker <function-name>`). CI builds and publishes images tagged with each function's semantic version (managed by `@jscutlery/semver`) to the Gamesys Artifactory registry.

## Deployment & CI

Deployments go through the GitLab CI pipeline (Pipelines → Run Pipeline) using the `CHANGES` and `ENVS` variables to target specific lambdas/layers and environments (e.g. `stg_eu00`, `prod_eu00`). Commits follow the [Conventional Commits](https://www.conventionalcommits.org/) spec, enforced by commitlint pre-commit hooks, which also drives semantic versioning per lambda.
